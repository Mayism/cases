# 课堂签到 需求PRD

> 版本：1.0.0 | 平台：HarmonyOS | 更新日期：2026-04-28

## 1. 业务场景

高校课堂考勤管理场景。任课老师在课堂上公布四位数字签到码，学生通过应用内的签到功能输入签到码完成考勤。系统获取学生当前位置信息用于验证签到有效性，确保学生在课堂范围内完成签到。

## 2. 业务目标

- 提供便捷的课堂签到入口，学生可在首页一键进入签到流程
- 通过位置信息验证提升签到真实性
- 支持桌面 Widget 快捷跳转签到，减少操作步骤
- 实时反馈签到状态（成功/失败/已签到），提升用户体验

## 3. 功能需求

### 3.1 首页签到入口

- 在首页 Banner 下方展示"去签到"按钮
- 按钮采用橙色渐变背景（#FFED6F21 → #FFFFC39F），包含签到图标和文字
- 右侧显示圆形箭头按钮，引导用户点击
- 点击行为：
  - 已登录用户：跳转到签到页面
  - 未登录用户：Toast 提示"请关联账号后使用签到"
  - 网络异常：Toast 提示"网络繁忙，请稍后重试"

### 3.2 签到页面

- 顶部标题栏显示"课堂签到"，支持返回按钮
- 顶部区域使用橙色到背景色的渐变效果
- 主体区域包含：
  - 签到码输入背景装饰图
  - 签到状态展示区（未签到时显示"点击输入四位数签到码"；已签到时显示课程名称和签到成功图标）
  - 四位数字输入框（支持逐位输入、自动跳转焦点、闪烁底线指示）
  - 签到说明文字
  - "确定"提交按钮（四位签到码输入完整后可点击）
- 签到码错误时：输入框边框变红、整体左右抖动、显示"请输入正确的签到码"
- 加载状态：全屏半透明遮罩 + LoadingProgress 指示器

### 3.3 桌面 Widget 卡片

- Widget 卡片展示两个按钮："首页"和"打卡"
- 点击"首页"：跳转到应用首页
- 点击"打卡"：直接跳转到签到页面

### 3.4 位置权限

- 进入签到页面时自动请求位置权限
- 首次请求被拒绝后支持二次弹窗确认
- 提交签到时需获取当前位置（经纬度）
- 若定位服务未开启，引导用户到系统设置开启

## 5. 核心技术方案

- **模块架构**：独立 HAR 模块（module_secure_checkin），通过路由表注册页面，entry 模块依赖引用
- **UI 框架**：ArkUI @ComponentV2 + @Builder，RelativeContainer 相对布局
- **状态管理**：@ObservedV2 + @Trace + @Computed 响应式模式
- **导航**：NavPathStack 路由栈，通过路由名称进行页面跳转
- **位置服务**：@kit.LocationKit geoLocationManager
- **权限管理**：abilityAccessCtrl 封装权限请求流程

## 6. 数据流向

```
首页(HomeTab)
  │
  ├── onClick "去签到" → HomeTabVM.goToCheckinPage()
  │     └── routerStack.pushOrPopToName(CHECKIN_PAGE_NAME)
  │
  └── Widget "打卡" → IntentTask.GO_TO_CHECKIN_PAGE
        └── IntentTaskVM.handleIntentTask()
              └── HomeTabVM.goToCheckinPage()

签到页面(CheckinPage)
  │
  ├── 初始化 → CheckinPageVM.initialize()
  │     ├── PermissionUtils.requestLocationPermissions()
  │     └── Api.getCheckinInfo() → 更新 state/courseName/isCheckedIn
  │
  ├── 输入签到码 → UIVerifyCodeInput
  │     └── onCodeChange/onFinishInput → CheckinPageVM.checkinCode
  │
  └── 点击"确定" → CheckinPageVM.doCheckin()
        ├── PermissionUtils.requestLocationPermissions()
        ├── geoLocationManager.getCurrentLocation()
        ├── Api.checkin({ courseId, checkinCode, location })
        │     ├── 成功 → isCheckedIn = true → 显示成功界面
        │     └── 失败 → showErrorTips = true → 输入框红框抖动
```

## 7. 交互流程

1. 用户在首页点击"去签到"（或通过 Widget 点击"打卡"）
2. 未登录 → Toast 提示"请关联账号后使用签到"，流程结束
3. 已登录 → 跳转到签到页面
4. 页面加载：请求位置权限 → 获取签到信息
5. 显示当前课程名称和四位签到码输入框
6. 用户逐位输入签到码（纯数字）
7. 输入完成 4 位后"确定"按钮变为可用
8. 用户点击"确定"
9. 系统获取当前位置 → 提交签到请求
10. 签到成功：显示课程名称 + 成功图标 + "签到成功"文字
11. 签到失败（签到码错误）：清空输入、红框抖动动画、错误提示
12. 用户可修改签到码重新提交

## 8. 功能边界与限制条件

- 签到码固定为 4 位纯数字
- 提交签到必须获取位置权限和位置信息
- 当前为 Mock 模式，签到码默认为 `1111`
- 签到码错误无重试次数限制
- 已签到状态不可重复签到
- Widget 跳转需应用已在后台运行或冷启动
- 位置精度优先级：FIRST_FIX（首次定位，速度优先）
- 定位超时时间：10 秒

## 9. 验收标准与成功指标

- 功能完整性：首页"去签到"按钮正常展示，点击后正确跳转
- 登录校验：未登录用户点击签到提示关联账号
- 签到页面：四位签到码输入交互正常，焦点自动跳转
- 签到提交：输入正确签到码后签到成功，页面状态更新
- 错误处理：签到码错误时输入框红框抖动并提示
- 权限管理：位置权限请求流程正常，拒绝后有合理提示
- Widget：桌面卡片"打卡"按钮可跳转到签到页面
- 编译成功：`hvigorw --mode project -p product=default assembleApp --analyze=normal --parallel --incremental --daemon`
